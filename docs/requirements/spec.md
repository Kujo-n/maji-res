# 要件定義書：MAJIレスシステム (MVP)

## 1. サービス概要

ユーザーの意見や意思決定案に対し、役割の異なるサブエージェントが「マジレス（忖度のない客観的指摘）」を行い、意思決定の精度を高めるための深掘り支援アプリ。

## 2. システムアーキテクチャ

設定ファイル（`config.json`）で定義される分析エージェント群と、統合層（Integrator）による合議構成。

### 2-1. エージェント構成

| レイヤー | エージェント名 | 役割         | 思考ロジック                                  |
| :------- | :------------- | :----------- | :-------------------------------------------- |
| 分析層   | MELCHIOR       | 肯定的視点   | メリット最大化、機会損失回避、促進的提案。    |
| 分析層   | BALTHASAR      | 批判的視点   | 論理的欠陥の指摘、リスク抽出、失敗シナリオ。  |
| 分析層   | CASPER         | 中立的視点   | 社会的通念、倫理性、メタ認知的な俯瞰。        |
| 統合層   | Integrator     | 論理統合・審議 | 分析層の整合性確認、不足情報の特定、合議判定。 |

> **Note**: エージェントの名前・役割・プロンプトはプリセットの `config.json` で定義されており、
> コード変更なしにカスタマイズ可能。詳細は「2-2. 設定可能化」を参照。

### 2-2. 設定可能化

エージェント定義はプリセットフォルダ方式で管理される。

- **プリセットフォルダ**: `src/lib/agents/prompts/<プリセット名>/`
- **切り替え方法**:
  1. **UI**: サイドバーの「SETTINGS PRESET」から選択（優先・ブラウザに保存）
  2. **環境変数**: `AGENT_PRESET` でデフォルトを指定
- **設定ファイル**: 各プリセット内の `config.json` で以下を定義:

| フィールド | 説明 |
| :--- | :--- |
| `provider` | LLMプロバイダー (`google` / `openai` / `anthropic`) |
| `defaultModel` | 全エージェント・Integratorが使用するデフォルトモデル |
| `agents[].id` | エージェントの一意識別子 |
| `agents[].name` | 表示名 |
| `agents[].role` | 役割の説明 |
| `agents[].promptFile` | 同フォルダ内のプロンプトMarkdownファイル名 |
| `agents[].model` | エージェント個別のモデル指定（`defaultModel`を上書き） |

## 3. 機能要件

### 3-1. 審議・合議ロジック

- 合意形成ルール（多数決）:
  - 判定: 各エージェントから「APPROVE」「DENY」「CONDITIONAL」の投票を収集し、多数決で `verdict` を決定。
  - スキップ機能: 意見が割れた（CONDITIONALが多数、またはAPPROVEとDENYが同数）場合は、統合エージェントによる結論の合成処理をスキップし、クラリフィケーション（追加質問）のみを即座にユーザーに提示する。
- コンテクスト管理:
  - 「意思決定台帳（Decision Ledger）」方式。毎ターンの合意・未決事項を統合要約して次ターンへ継承。
- 保持期間：原則制限なしとするが、システムリソース保護のためユーザー単位での上限を設ける。
- 上限設定：
  - スレッド数：管理者50、一般ユーザー20
    - 上限を超える新しいスレッドを作成しようとした場合は、最も更新日時が古いスレッドが自動的に削除される。
  - メッセージ数（1スレッドあたり）：管理者100、一般ユーザー50
    - 上限を超える場合はエラーとなり、それ以上の対話はできないため、新しいスレッドの作成を促す。
  - これらの設定は環境変数（`NEXT_PUBLIC_ADMIN_THREAD_LIMIT`など）により変更可能。

### 3-2. 深掘り（逆質問）アルゴリズム

- 目的: 手段（How）の前に、目的（Why）と成功の必須要素を問い直す。
- 挙動: 分析層の意見が対立した「ピボットポイント」を特定し、判断に必要な情報をユーザーに逆質問する。
- UI挙動:
  - 各質問に「回答する」ボタンを表示。クリックでチャット入力欄に `「{質問}」への回答: ` をプリセットし、自動フォーカス。
  - 質問が2件以上の場合は「すべてに回答する」ボタンを追加表示。クリックで番号付きテンプレートを一括プリセット。

### 3-3. 例外処理（ちゃぶ台返し対応）

- 矛盾検知: 過去の確定事項と矛盾する入力を検知した場合、Integratorが「自覚の確認」をユーザーに行う。
- リセットフロー: 前提変更を認める場合、台帳を初期化しRound 1から再開。過去の議論は微かな「バイアス」としてのみ残す。

### 3-4. アクセス制限と権限管理 (Admin Portal)

- オープンアクセス防止: 無制限なLLMリクエストを防ぐため、システム利用を事前承認制とする。
- 新規登録フロー:
  - 未登録ユーザーがGoogleアカウントでログインした場合、Firestore上に `status: pending` として一時保存され、アプリへのアクセスは拒否される。
  - 同時に、Gmail SMTPを介して管理者（指定メールアドレス）へ承認依頼の自動通知メールが送信される。
- 管理用ダッシュボード (`/admin`):
  - `role: admin` 権限を持つユーザーのみがアクセス可能。
  - Pendingユーザーのリストを確認し、UI上から `APPROVE` ボタンをクリックすることで対象ユーザーのステータスを `active` へ変更する。

## 4. UI/UX要件（モバイル最適化）

### 4-1. 視覚演出

- レイヤード・スタック: 分析層の思考プロセスを垂直方向のリアルタイム・ストリームで表示。生成完了後に一括表示するのではなく、テキストが順次タイピングされる様子を視覚化し、ユーザーの待機時間を心理的に軽減。
- ステータスバー: 固定ヘッダーにエージェントの同調状態（シンクロ率）を六角形アイコンで表示。
- 判定演出: 可決/否決時に全画面オーバーレイとハプティクス（振動）によるフィードバック。

### 4-2. 回答フロー（シンクロ・エディタ）

- コンテクスト・ドロワー: 入力中にエージェントの過去の発言をオーバーレイ参照・引用可能。
- ハイブリッド・入力: 自由記述に加え、数値やトグルによる構造化データを収集。

- Default: 機械的、箇条書き、感情を排したレポート形式。

## 6. 利用環境要件

本システムは、以下の環境においてシームレスに動作することを要件とする。

- PC環境:
  - 3カラム表示による情報の同時並行的な視認性を最大化する。
  - キーボードショートカットによる迅速なコンテキスト引用・操作をサポートする。
- スマートフォン（モバイル）環境:
  - 縦長画面に最適化した「レイヤード・スタック構造」を採用する。
  - ハプティクス（振動）やスワイプジェスチャーを駆使した直感的な操作感を実現する。
  - 外出先での隙間時間における「思考の壁打ち」を想定した軽量なUIレスポンスを維持する。

## 7. 客観的分析と予測（事実と予測）

### 【事実】

- マルチエージェントの有効性: LLMの役割を分散させることで、単一プロンプトよりも論理性と事実確認の精度が向上する（Chain-of-Debate手法）。
- 情報のボトルネック: 長大な履歴はLLMの精度を下げるため、統合ステート（台帳）方式は長期的な議論において計算効率と精度のバランスに優れる。

### 【予測】

- ユーザー心理: 「マジレス」という厳しいフィードバックに対し、ユーザーは「自分の案が真剣に吟味されている」という心理的充足を得やすく、リテンション向上に寄与する。
- モバイル体験: リアルタイムな判定演出や、各エージェントの思考過程のストリーミング表示により、通信環境やLLM生成遅延に伴う体感的な待機時間が大幅に緩和され、継続的なインタラクションを促進する。

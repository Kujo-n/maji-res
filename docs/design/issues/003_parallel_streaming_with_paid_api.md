# Issue 003: 有償API（Gemini Pro等）への切り替えとエージェント並列処理化の対応方針

## 概要
現在、MAJIレスシステムの分析層エージェント（各プリセットで定義される3エージェント）は、Gemini Flash APIの無償枠における厳しいレートリミット（RPM制限等）を回避するため、意図的に直列処理（各エージェントの実行間に2000msの待機時間を設ける）で実装されています。

将来的に有償枠API（例: 有償版Flashや上位モデル）へ移行した際には、このレートリミット制限が緩和されるため、エージェント処理を完全に並列化し、レスポンスタイム全体を大幅に短縮することが可能になります。本チケットでは、その際の対応方針を整理します。

## 現状の実装（2026-03時点）

### アーキテクチャ概要
- **プリセット機能**: エージェント構成はプリセット単位（`MAGI`, `MAJI-RES`, `MATCHGRAM`）で `config.json` に定義され、`AgentIntegrator` がプリセットに応じてエージェントを動的に生成する。
- **エージェント構成**: `ConfigurableAgent` クラスにより、プリセットごとに異なるペルソナ・プロンプトを持つエージェントが構成される。マルチプロバイダ対応（Google, OpenAI, Anthropic）は `provider-resolver.ts` で実現済み。
- **モデル**: 全プリセットのデフォルトモデルは `gemini-2.5-flash`（`config.json` で定義）。
- **ストリーミング**: `streamSynthesize` によるリアルタイムストリーミング統合、レートリミットのリトライロジック（最大3回、指数バックオフ）が実装済み。
- **フロントエンド**: `useMagiChat` フックは `useChatSession`（状態管理）と `useMagiStream`（ストリーム通信・パース）に分割済み。`LayeredStack` はカスタムペルソナ対応の汎用色分けに対応。
- **APIルート**: `src/app/api/magi/stream/route.ts` にて、認証・レートリミット・プリセット指定・ストリーミングを一元管理。

### 直列処理の現状
`src/lib/agents/integrator.ts` の `parallelProcess` メソッドにおいて、以下の直列処理ロジックが残存:
- `for` ループによる順次実行
- 各エージェント処理完了後に `setTimeout(2000ms)` で待機
- 各エージェントのストリーミングチャンクは `onAgentUpdate` コールバックでリアルタイムにフロントへ通知

## 対応方針・予定される変更箇所

### 1. `src/lib/agents/integrator.ts` の並列化
現状の `parallelProcess` メソッド内にある直列ループ処理と `setTimeout` による待機（2000ms間隔）を廃止します。代わりに `Promise.all` 等を用いて、全エージェントのAPI呼び出しを同時にキックする完全パラレル処理に書き換えます。各エージェントのストリーミングチャンクは、並列に受信しながら `onAgentUpdate` コールバックで随時フロントへ通知する設計とします。

### 2. 環境変数およびモデル設定の変更
API利用形態の変更に合わせて、システムが使用するモデル設定を更新します。
- 各プリセットの `config.json`（`src/lib/agents/prompts/<PRESET>/config.json`）にてデフォルトモデルを必要に応じて上位モデルへ切り替えます。
- `provider-resolver.ts` のマルチプロバイダ対応により、プリセット単位でプロバイダ・モデルの変更が可能です。
- 課金状況やトークン利用量（Usage）をモニタリングできる仕組みを導入・確認します。

### 3. フロントエンドの並列ストリームレンダリング検証
バックエンドが並列でストリームを返却し始めた場合、`useMagiStream` は複数エージェントのテキスト差分を同時に受信することになります。
現状のUI（`LayeredStack` でのカード表示など）において、複数のカードが同時に更新されても負荷やレンダリングの乱れが生じないかを確認・最適化します。

### 4. レートリミットリトライロジックの見直し
`streamSynthesize` に実装済みのレートリミットリトライ（指数バックオフ）について、有償枠移行後に不要となる場合は簡素化、または設定で無効化できるようにします。

## 備考
- 本チケットは、サービスの利用者増加や運用要件によりAPIレスポンス速度の向上が必須となったタイミングで着手します。
- コストとパフォーマンスのトレードオフを監視し、段階的（まずは有償枠のFlashで上限解放＋並列化、その後必要であればProモデルへ移行）に進めることも可能です。
- プリセット単位でモデル・プロバイダを切り替えられるため、特定プリセットのみ先行して有償モデルへ移行するアプローチも検討可能です。

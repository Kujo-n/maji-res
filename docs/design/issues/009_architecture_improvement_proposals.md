# Issue 009: フロントエンドアーキテクチャの改善提案（将来の課題）

**Date:** 2026-03-07
**Status:** Open (Pending Implementation)
**Related Persona:** `web_architect.md`

## 概要

MAJI-RES アプリケーションのフロントエンドアーキテクチャは現在、React/Next.jsエコシステムを適切に活用し「関心の分離（SoC）」が綺麗に保たれた優良な状態です。
しかし、中長期的なスケーラビリティ、パフォーマンス、安定性をさらに一歩高めるため、以下の改善案を将来の課題（ISSUE）として記録します。現在は実装を行わず、必要になったタイミングで着手します。

## 改善提案詳細

### 1. Zod による Strict な入力・出力検証
- **課題**: 現在、Firestore（`ChatService`等）への保存時や、LLMからのストリーミングレスポンスのパース時において、厳格な型検証（ランタイムでの検証）が一部手薄な箇所があります。
- **提案**: 
  - ユーザーからのチャット入力や、バックエンドAPI/Firestoreへ送受信するデータモデルに対して `Zod` スキーマを定義し、パース/バリデーションを徹底する。
  - これにより、未定義データ（`undefined` 等）の混入や不正な値によるサイレントなバグをコンポーネント外部で早期に防ぐことができます。

### 2. パフォーマンス・レンダリングの最適化
- **課題**: チャット機能（`useChatSession` 等）は状態更新が頻繁に発生します。現状は問題ありませんが、メッセージ履歴が非常に長くなった場合や、コンテキストの再計算時に不要な再レンダリングコストが生じる懸念があります。
- **提案**:
  - React DevTools の Profiler 等を用いてレンダリング回数を計測する。
  - ボトルネックとなるコンポーネント（長大なリストアイテム等）に対して `React.memo` を適用する。
  - 複雑な派生状態の計算に対して `useMemo` やコールバック関数に `useCallback` を明示的に追加する。
  - （または React Compiler などの導入を検討する）

### 3. サーバーコンポーネント (RSC) のさらなる活用
- **課題**: 現在多くのUIコンポーネント（チャットUIやサイドバー等）で `"use client"` が使用されていますが、初期ローディング時に静的なデータ取得部分をもう少しサーバー側に寄せられる可能性があります。
- **提案**:
  - アプリケーション全体のレイアウトや、初期のルーティング・データフェッチングにおいて、Server Component で処理可能な部分を洗い出してリファクタリングする。
  - クライアントに送信するJavaScriptバンドルサイズを削減し、初期ロード（FCP/LCP）の高速化を図る。

## 今後のアクション

- 本 ISSUE は中長期的な技術的負債の予防策として位置づけます。
- 新機能開発が落ち着いたタイミング、またはパフォーマンスに関する具体的な課題（動作の重さなど）が顕在化した際に、優先度を見直して着手することとします。
